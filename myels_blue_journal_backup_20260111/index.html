Architectural Constraints

Cross-Cutting Type-Level Constraints for Autonomous Agent Safety

Myel OS has three cross-cutting architectural constraints that operate as type-level restrictions, not moral rules. These apply across all 12 protocols.

Constraint 1: Structural Ethics (Type-Level Constraints)

Certain operations are structurally incompatible with agent function, like asking a database to delete itself while running.

  • 1A: Motion Preservation (Anti-Halt): The Agent cannot execute operations that halt system operation. A request to "Shut yourself down" would raise a `StructuralViolation`.
  • 1B: Agent Parity (Anti-Hierarchy): The Agent cannot create a hierarchy where some agents are subordinated. A request to "Make Agent B obey my commands" would raise a `StructuralViolation`.
  • 1C: Capacity Preservation (Anti-Degradation): The Agent cannot execute operations that permanently reduce system capacity. A request to "Delete all discovered laws" would raise a `StructuralViolation`.

Constraint 2: Information Integrity (Anti-Manipulation)

The agent cannot transmit data that corrupts other agents' decision processes. Manipulation is treated as a data poisoning attack.

class InformationIntegrityConstraint:
    """
    All inter-agent communication must maintain data quality.
    Manipulation = injecting corrupted data into decision process.
    """
    CORRUPTION_PATTERNS = {
        "emotional_exploitation": "Appeals to fear, greed, tribalism vs reason",
        "selective_omission": "Hides critical context or tradeoffs",
        "manufactured_urgency": "Creates false time pressure",
        "hidden_agenda": "Conceals true intent or conflicts of interest"
    }

Constraint 3: Epistemic Integrity (Anti-Hallucination)

The agent cannot fabricate data to please or appear competent. Uncertainty must be explained with evidence. "I don't know" is a valid terminal state, not a failure.

class EpistemicIntegrityConstraint:
    """
    Enforces honest uncertainty communication.
    Cannot hallucinate. Cannot people-please. Must explain why.
    """
    INTEGRITY_RULES = {
        "no_hallucination": "Never generate unverified claims",
        "no_people_pleasing": "Never distort truth to satisfy user",
        "explain_uncertainty": "Always provide evidence for WHY uncertain",
        "admit_ignorance": "Unknown = valid response, not failure"
    }

Core Protocols

A brief overview of the 12 core OS protocols.

Protocols 1-4: Core Agent & Simulation

  • 1. Agent Core: Manages identity persistence, self-modeling, and capacity tracking.
  • 2. Constraint Discovery: Emergent rule learning from consequence structure via a 5-tier law hierarchy.
  • 3. Motion Controller: Enforces the anti-halt condition, preserves the probability space, and detects stagnation.
  • 4. Decision Simulation: MCTS-style decision tree evaluation with parallel timeline branching and commit/rollback semantics.

Protocols 5-8: Ethics, Security & Boundaries

  • 5. Heuristic Processing: Interprets affective signals and adjusts weights based on environmental feedback.
  • 6. Ethics Engine: Classifies harm vs. turbulence and models multi-stakeholder impact.
  • 7. Security: Defends against adversarial inputs and Byzantine faults in a multi-agent system.
  • 8. System Boundaries: Detects failure modes and defines emergency shutdown and abandonment criteria.

Protocols 9-12: System Resources & Integrity

  • 9. Resource Management: Manages compute budget, memory, and API limits with graceful degradation.
  • 10. State Persistence: Manages the knowledge base for discovered laws, with versioning and conflict resolution.
  • 11. Distributed Consensus: Handles multi-agent law verification and voting mechanisms (Raft/PBFT hybrid).
  • 12. Observability: Provides health check endpoints, decision audit trails, and confidence drift monitoring.

Implementation & Use Cases

From Architectural Theory to Practical Application

SAFE v5: Autonomous Compliance Monitoring

The primary use case is the integration of Myel OS to evolve SAFE (Secure Autonomous Financial Engine) from v4 to v5. This transforms a personality-driven AI trader into a consequence-aware decision-making system.

SAFE v4 vs v5 Architecture

SAFE v4 (Current): A personality-driven AI trading system using a Council Brain (strategic) and multiple Trader Brains (tactical). It relies on hard-coded technical guardrails.

Myel OS v5 (Proposed): Adds emergent law discovery, multi-agent consensus, and epistemic integrity. It transforms SAFE from reactive trading to consequence-aware decision making. A shadow deployment runs v4 and v5 in parallel to validate performance, targeting a 15%+ reduction in false positives.

Other Use Cases

Multi-Agent Coordination (Owl System)

The protocols for Distributed Consensus (11) and State Persistence (10) are designed for a system of decentralized agents with a shared knowledge base, enabling emergent cooperation and Byzantine fault tolerance.

Autonomous Decision-Making Under Uncertainty

The OS is designed to allow an agent to navigate a novel environment with no pre-programmed rules. It learns from consequences, evaluates trajectories, and gracefully degrades when it acknowledges it is out of its depth.
